# TODO 도메인 인터뷰 전사 정리

Source: Discord audio transcript summarized for the TODO domain
Updated: 2026-05-23

> 자동 전사 기반 정리다. 고유명사와 일부 발화는 오인식 가능성이 있으므로 PRD에는 확인된 요구사항 중심으로 반영한다.

## 1. TODO 핵심 문제

1. TODO를 만들어도 잊어버린다.
2. 지난 일정이나 못 한 일을 사용자가 직접 하나씩 미뤄야 한다.
3. 일정 확인 자체가 부담이 된다.
4. 업무 배분과 진행도 확인을 Google Sheet 링크 공유에 의존한다.
5. 개인 작업, 공동 작업, 사무국 작업, 전체 공유 작업이 직관적으로 구분되지 않는다.
6. Discord로 업무를 모으고 싶지만, 텍스트 입력만으로는 Google Calendar/Sheet보다 불편해질 위험이 있다.

## 2. 현재 사용 방식

| 영역 | 현재 방식 | 메모 |
| --- | --- | --- |
| 개인 일정 | Google Calendar, Google Tasks | 총무 개인 기준으로 관리 |
| 사무국 문서 | Google Docs, Google Sheets | 주로 총무가 만들고 사무국이 확인 |
| 공유 일정 | KakaoTalk Calendar | 전체 참여자 접근성이 높아서 사용 |
| TODO/업무 분담 | Google Sheet 링크 공유 | 진행도 확인은 가능하지만 접근 장벽이 있음 |
| 자료 아카이브 | Google Drive 후보 | 향후 첨부자료/영수증 저장소로 고려 |

## 3. 우선 해결 요구

1. 오늘 못 한 TODO를 `내일`, `3일 뒤`, `직접 지정 날짜`로 쉽게 미룬다.
2. 지연 처리는 완전 자동보다 사용자가 선택하는 반자동 방식으로 시작한다.
3. 총무가 구성원에게 업무를 직접 배분하되, 배분/확인 조작을 단순화한다.
4. 담당자별 TODO 목록과 공동 작업 목록을 볼 수 있게 한다.
5. Discord 안에서 TODO 생성, 조회, 상태 변경, 지연 처리를 처리한다.
6. 알림 대상은 공개 범위와 권한에 따라 분리한다.

## 4. 역할과 공개 범위

### 4.1 역할 필드

| 필드 | MVP 필수 | 설명 |
| --- | --- | --- |
| 담당자 | yes | 실제 처리 책임자. 현재 당장 필요한 핵심 필드 |
| 공동 작업자 | no | 함께 진행하거나 확인할 사람 |
| 확인자 | no | 결과 확인 담당. 확장용 |
| 요청자 | no | 일을 요청한 사람. 확장용 |
| 생성자 | yes | TODO를 입력한 사람 |

### 4.2 작업 범위

1. 개인 작업: 본인에게만 보이는 TODO/일정.
2. 사무국 작업: 사무국 채널에서 공유되는 TODO/일정.
3. 공동 작업: 여러 담당자 또는 역할이 연결된 TODO.
4. 전체 공유 작업: 전체 채널에 알림이 필요한 일정/업무.

## 5. 알림 정책

| 범위 | 알림 위치 |
| --- | --- |
| 개인 일정/TODO | 개인 DM |
| 사무국 일정/TODO | 사무국 채널 |
| 기청 참여자 전체 일정 | 전체 채널 |

알림 피로도를 줄이기 위해 카테고리, 공개 범위, 권한 기반으로 수신자를 나눈다.

## 6. Discord TODO MVP 후보

1. `/todo add`: TODO 생성.
2. `/todo list`: 오늘/미완료 중심 목록 조회.
3. `/todo status`: wait/progress/done/dismiss 상태 변경.
4. `/todo delay`: 미완료 TODO를 내일/며칠 뒤/지정 날짜로 미룸.
5. `/todo assign`: 담당자와 공동 작업자 지정.
6. `/todo remind`: 공개 범위별 알림 예약.

## 7. Google Workspace/LLM 연동 방향

1. Google Sheets는 초기 저장소 또는 운영자가 보기 쉬운 보조 화면 후보로 유지한다.
2. Google Calendar/Tasks 연동은 MVP 이후 검토한다.
3. LLM/음성 입력은 텍스트 기반 Discord UX를 보완하는 강한 후보지만, 저장 전 확인 단계가 필요하다.
4. Gemini/ChatGPT 등 provider 선택은 비용, Google Workspace 제어 가능성, 범용성을 비교해 결정한다.

## 8. PRD 반영 결정

1. TODO MVP는 지도/이동 시간보다 `잊어버림`, `수동 지연`, `업무 배분`, `공동 작업 가시성`, `알림 분리`를 우선한다.
2. 일정 기능은 TODO의 due/reminder 범위에서 시작하고, 완전한 캘린더 대체는 비목표로 둔다.
3. 자동 지연은 MVP에서 제외하고, 사용자가 선택하는 반자동 지연을 제공한다.
4. Google Workspace 연동은 중요한 확장 방향이지만 첫 구현은 Discord command 계약과 fake/local storage로 검증한다.

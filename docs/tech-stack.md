# Activities Tech Stack

## Scope

This document defines the MVP technical stack for the local Discord bot.

The existing React/Spring/PostgreSQL scaffold remains in the repository, but it is not the first implementation target. The first target is a Docker Compose-managed local bot that serves one Discord server with service-specific channels:

| Channel | Service |
| --- | --- |
| `#sns` | SNS upload |
| `#receipt` | Receipt and transfer status |
| `#todo` | Todo and schedule management |

## MVP Stack

| Layer | Choice | Notes |
| --- | --- | --- |
| Bot runtime | Node.js + TypeScript | Shared runtime for Discord command handlers. |
| Discord SDK | `discord.js` | Slash commands, modals, buttons, attachments. |
| Google Sheets API | `googleapis` | People Sheet, Transfers Sheet, service logs. |
| Local state | SQLite | Bot config, sheet URLs, command metadata, local file metadata. |
| Local file storage | Filesystem | Uploaded receipts and service files. |
| Process/runtime | Docker Compose | Local operation and restartable service definition. |

## Local File Storage

Receipt images are stored under this path:

```text
data/receipts/{yyyy}/{MM}/{generated_filename}
```

Rules:

1. `yyyy` and `MM` come from the receipt upload time.
2. `generated_filename` must be generated by the bot.
3. User-provided filenames must not be used directly in paths.
4. The original filename may be stored as metadata in SQLite or Google Sheets.
5. The bot must create missing year/month directories.
6. If the storage directory is not writable, receipt upload must fail before writing to Google Sheets.

## SQLite Responsibilities

SQLite stores local bot state that should not depend on Discord messages or Google Sheets formatting.

Minimum tables:

- `bot_config`
  - `key`
  - `value`
  - `updated_by`
  - `updated_at`
- `local_files`
  - `id`
  - `service`
  - `local_path`
  - `original_filename`
  - `mime_type`
  - `size_bytes`
  - `created_by`
  - `created_at`
- `command_audit_log`
  - `id`
  - `service`
  - `command`
  - `discord_user_id`
  - `discord_channel_id`
  - `status`
  - `created_at`

See `docs/data-schema.md` for the full MVP schema.

## Failure Handling

External operations should use a small retry loop before failing the user flow.

Rules:

1. Google Sheets read/write, local file save, SNS upload, and Discord message update may retry 2 to 3 times.
2. After the final failure, the bot returns a user-readable message and ends the flow.
3. Retries must be idempotent. They must not create duplicate Sheets rows, duplicate local files, or duplicate SNS posts.
4. Internal errors should be logged to SQLite `command_audit_log`.
5. Discord responses must not expose tokens, credentials, stack traces, or local absolute paths.

## Docker Compose Responsibilities

Docker Compose should run the local bot with mounted data directories.

Expected mounted paths:

```text
./data:/app/data
```

Expected environment:

```text
DISCORD_TOKEN=
DISCORD_CLIENT_ID=
DISCORD_GUILD_ID=
GOOGLE_APPLICATION_CREDENTIALS=/app/data/google-service-account.json
SQLITE_PATH=/app/data/activities.sqlite
RECEIPT_STORAGE_DIR=/app/data/receipts
```

## Deferred Stack

These remain deferred until the Discord bot proves real usage:

- React dashboard.
- Spring Boot API expansion.
- PostgreSQL product persistence.
- S3 or cloud object storage.
- Hosted multi-tenant deployment.
- KakaoTalk automation.

## Open Decisions

1. Whether service data should eventually move from Google Sheets/SQLite into PostgreSQL.
2. Whether SNS uploaded assets should share the receipt filesystem pattern.
3. Whether Todo requires a web/mobile companion UI for long lists and map-heavy flows.
